# term-drift の理論的背景

[English](theory_en.md) | 日本語

term-drift は、AI 支援開発でプロジェクトへ流入した用語のずれを見つけ、**人が確認したか、明示委任された範囲で低リスクと判断された言い換えだけ**を安全に適用するための軽量な仕組みです。

ここで扱う問題は、単なる表記ゆれではありません。同じ概念に複数の呼び名が増えること、普通の言葉が内輪の特殊な意味へ転用されること、定義は存在しても使用箇所から辿れないことを含みます。これらが積み重なると、文書は文法的には読めても、初めて参加した人には意味が復元できなくなります。

term-drift はこの問題を、Domain-Driven Design の **Ubiquitous Language（ユビキタス言語）**、用語管理、認知負荷、common ground（共有された理解）、情報探索、Human-in-the-Loop の考え方を組み合わせて扱います。意味判断を完全自動化せず、LLM・人・決定的なプログラムへ役割を分けるのが中心的な設計です。

**読まなくても使えます。** 実際の手順は [`rules/workflow.md`](../rules/workflow.md) にあります。この文書は「なぜ台帳が要るのか」「なぜ辞書照合だけでは足りないのか」「なぜ判断権限を明示するのか」を確認したいときの参照用です。

## 問題 — AI は「言葉を大量に持ち込む参加者」である

開発者同士でも用語は自然に揺れます。しかし AI コーディングエージェントは、短時間に大量の設計文書・コミットメッセージ・説明文を生成するため、揺れの**速度と複製範囲**を大きくします。

ここでいう **term drift** は、自己適用と範囲を限定した監査で観測した問題をまとめるための本ツールの作業概念です。Furnas らの vocabulary problem などは構成要素を説明しますが、「AI が用語ずれを増幅する」という現象全体を直接測定した研究として引用しているわけではありません。

モデルは一般コーパス、他プロジェクトの慣用、直前の会話からもっともらしい語を選びます。その語が対象プロジェクトで合意済みかどうかは、言語モデルの語彙知識だけでは決まりません。局所的には自然な表現でも、プロジェクト全体では次のずれを作り得ます。

- 既存概念に別名を付け、検索と会話の入口を分裂させる。
- 一般語を特殊な意味に転用し、初見の読み手に二重の解釈負荷を与える。
- 一度だけ説明した語を参照なしで広げ、定義へ辿れない状態にする。
- 生成された語を次の生成が引用し、「多く使われているから正しい」という見かけを作る。

Furnas らの “The Vocabulary Problem” が示したように、人は同じ対象へ同じ語を選ぶとは限りません。AI はこの古典的な呼称不一致を消すのではなく、高い生成速度によって増幅し得ます。

term-drift が扱う **term drift** は、このような「語の形・意味・定義への経路が、チームの合意からずれること」です。識別子の改名や外国語の訳語選定一般は対象にしません。grep や実行手順で参照されるコマンド名・関数名・設定キーは機械的な契約でもあるため、散文の言い換えと混ぜないからです。

## 正規語彙は辞書ではなく、チームの社会的な契約である

Eric Evans の Ubiquitous Language は、専門家と開発者がモデルを語るために共有し、会話・文書・コードで継続して使う言葉です。重要なのは「一般の辞書に載っているか」ではなく、**このチームで同じ意味として合意されているか**です。

したがって term-drift は、語の身分を語そのものから推定しません。次の3分類を、台帳の状態と読み手に相対して決めます。

1. **一般語** — 想定した読み手の基礎語彙に入る語。
2. **チーム共通語** — 台帳に承認済みで載る語。
3. **未承認の独自用語の疑い** — 台帳に無い語、暫定の語、または一度否認されたのに再び使われた語。

この分類は Clark & Brennan の grounding の考え方と整合します。伝達が成立するには、送り手が語を知っているだけでなく、相手もその意味へ到達できるという共通基盤が必要です。「専門家なら分かる」「英語圏では普通」といった判断も、読み手が変われば結果が変わります。

台帳は真理の辞書ではなく、合意の記録です。そのため状態を持ちます。

- **承認済み** — チーム共通語として扱う。
- **暫定** — 誰かが使い始めたが、合意はまだない。
- **否認済み** — 採用しない。行は削除せず、同じ語の再発明を検出できるようにする。

状態未記載を承認済みにせず暫定へ倒すのは、既存文書に現れたという事実を、チームが承認した証拠として扱わないためです。

## なぜ3層で検出するのか

### 層1 — 台帳差集合

最初の層は、走査素材にあるプロジェクト固有らしい語と、台帳の正規語・別表記を照合することです。これは controlled vocabulary を使う用語管理の素直な適用です。

ただし、この層だけでは十分ではありません。台帳に無い固有名詞句は拾えても、`drain`、`settle`、`lobby`、`配線する` のように、辞書には普通の語として載る表現の特殊な使い方を見落とすからです。

### 層2 — 比喩転用と意味のずれ

Lakoff & Johnson が論じたように、比喩は飾りではなく、概念を理解し構造化する基本的な働きです。ソフトウェア開発でも `pipeline`、`drain`、`bridge` のような比喩は有用です。一方、比喩が局所的な意味を持ったまま説明なしで広がると、既知の一般的意味とプロジェクト内の意味を同時に保持させる二重負荷になります。

そのため第2層は、語が珍しいかではなく、**普通の意味からプロジェクト固有の意味へ転用されていないか**を読みます。これは単純な単語リストや正規表現では安定して判定できず、前後の文脈を読む層が必要になります。

### 層3 — 定義への到達可能性

語がどこかで定義済みでも、使用箇所からその定義へ辿れなければ、初見の読み手にとっては未定義とほぼ同じです。Pirolli & Card の information foraging では、人は手掛かり（information scent）を使い、読むコストに見合う情報源を探します。定義へのリンク・初出説明・用語集への参照がなければ、探索の手掛かりが弱くなります。

そこで第3層は、定義の**存在**だけでなく、使用箇所からの**到達可能性**を見ます。

| 状態 | 台帳差集合 | 比喩転用 | 到達可能性 |
|---|---:|---:|---:|
| 未登録の固有概念名 | 検出しやすい | 場合による | 定義があれば確認 |
| 普通の語の内輪転用 | 素通りしやすい | 主に検出 | 定義への導線も確認 |
| 定義は別文書にだけ存在 | 登録済みなら沈黙し得る | 場合による | 主に検出 |

3層は競合する別方式ではなく、異なる見落とし方を補う関係です。

## なぜ検出を完全な機械判定にしないのか

「一般語か」「実在の外部固有名詞か」「初見の読み手に説明が要るか」は、文字列だけでは決まりません。基礎語彙の固定リストを持てば、言語・職種・プロジェクトごとの保守が新しい問題になります。Web 頻度を正解にすれば、外部で多く使われる語と、このチームで合意された語を混同します。

term-drift はここを LLM の読解へ委ねます。既定では人が判断し、利用者が明示した範囲では LLM が低リスク箇所を判断できます。

```text
走査対象の収集 ──> 文脈を読んだ検出・分類・提案 ──> 人確認または明示委任
   決定的CLI                 宿主LLM                   人／宿主LLM
       │                                                   │
       └──────── 判断済み辞書 ──> 適用 ──> 再検査 <───────┘
                               決定的CLI
```

- **LLM** は候補と根拠を出し、明示委任された低リスク箇所を判断する。高リスク・意味不明は人へ確認する。
- **人** は委任範囲を決め、重要な意味判断と台帳変更を担う。
- **CLI** は判断権限が記録された明示データだけを、同じ入力なら同じ結果になる形で適用する。

この分業は Human-in-the-Loop の単なる飾りではありません。Bainbridge の “Ironies of Automation” が指摘するように、自動化が判断まで引き取るほど、人は監視に必要な注意と文脈を失いやすくなります。そこで重要な意味判断と委任範囲は人に残し、範囲内の低リスク判断と反復的な適用だけを機械へ渡します。

## 判断権限を明示し、意味が同じ箇所をまとめる

`approved: true` は「モデルの確信度が高い」という印ではありません。人が箇所を直接承認したか、明確な範囲をエージェントへ委任し、その範囲で低リスクと判断された書き換えであることを示す **authorization** です。

新規辞書はこの違いを自由記述の `note` だけに預けません。`decision_metadata_version: 1` を宣言し、各変更へ `decision_source`・`decided_at`・`delegation_scope` を構造化して記録します。人の個別承認は `human-approved`、委任内の判断は `delegated-agent` です。CLIは適用前に組み合わせを検証し、適用結果へ同じ情報を返します。辞書を `.term-drift/` に残すことで、本文へマーカーを混ぜずに後から判断元を辿れます。旧辞書は互換のため読めますが、判断元は `legacy-unknown` です。

既定では意味グループごとに人が判断します。包括委任は、語ごとに次の判断が異なるため推奨しません。

- 一般語としてそのまま残す。
- チーム共通語として台帳へ登録する。
- 普通の言葉へ言い換える。
- 特定のファイルだけ、理由を添えてそのまま残す。
- 意味を確認できず保留する。

利用者がリポジトリ、現在のレビュー、特定語など明確な範囲を委任した場合、エージェントは全出現の個別意味検査を続けながら低リスク箇所を判断できます。意味不明、重要な代案、義務・法務・セキュリティ・公開API・実行時挙動へ影響する箇所は人へ戻します。これにより質問の反復を減らしますが、検査や適用範囲の記録は省略しません。

同じ語の中でも、出現箇所ごとに主体・対象・因果・強さ・例外が異なります。全出現を棚卸しし、これらの意味要素、分類、書き換え方、意味保存理由が同じ箇所だけを同一グループにします。ガイド方式の承認は列挙された箇所に効きます。委任方式では、委任範囲で後から見つかった箇所も個別検査後に低リスクなら判断できます。

通常の判断画面では、グループ内の全ファイル・行と各引用・完成文を列挙し、共通の意味保存理由を1文で示します。代表例や件数だけで承認対象を隠しません。利用者はグループを分割でき、曖昧な箇所だけ説明を広げられます。承認後、CLIへ渡す辞書はグループを箇所単位の項目へ展開し、対象 `path` とそのファイル内で1箇所にだけ一致する文章を要求します。これにより対話上はまとめても、機械適用が裸の語による一括置換になることを防ぎます。

台帳への登録も同じです。暫定から承認済みへの昇格は分類器の出力ではなく、チームの言葉を変更する意思決定だからです。

## 適用は翻訳ではなく、安全な migration である

用語の言い換えは小さな文字列変更に見えますが、文書間の参照、コマンド例、識別子、過去の説明へ波及します。term-drift は適用を、データ migration に近い操作として扱います。

### 書き込み前の条件

- 対象は git 管理下で、ファイルが追跡済みである。
- git index と異なる未ステージ差分があっても、現在内容に承認済みの `from` が一意に一致する場合は、対象外の差分を保持して適用し警告する。曖昧一致や重なる書き換えは拒否する。
- UTF-8 として妥当でない文書は、再エンコードによるバイト破壊を避けるため触らない。
- Markdown の標準的なコードフェンス、単一行のインラインコード・HTML コメント・リンク先は機械参照として保護する。独自記法や壊れた Markdown まで完全に解析するパーサではない。
- 空の置換先、重複した置換元、連鎖して再適用時に結果が変わる辞書は拒否する。
- 新形式の辞書では、人承認または委任判断の構造化メタデータが欠ける項目を拒否する。
- 全対象を検証してから書き始め、辞書不正による途中適用を避ける。

事前検証は辞書不正による途中適用を防ぎますが、複数ファイルの書き込みをトランザクション化してはいません。書き込み中の I/O エラーやプロセス停止に対する自動ロールバックはなく、その場合の復元は git に依存します。

### 適用後の条件

- 承認した置換元が散文に残っていない。
- 再び適用しても変更が起きない（冪等）。
- 理由を添えて残したものだけが、例外として別枠に残る。
- 未検査・未適用の対象があれば成功扱いにせず、終了コードと `skipped*` で知らせる。

この構造は Design by Contract に近いものです。人が意味を承認したことが事前条件、適用後に残存がなく再適用で変化しないことが事後条件です。

## 例外として残すなら、理由も残す

どの用語にも例外はあります。電気工事の文書での「結線」のように、本来の意味で残すほうが正しい場合です。例外を禁止すると、利用者は検査そのものを無効化するようになります。

一方、理由を書かずに例外扱いすると、誤検出だったのか、判断を先送りしただけなのか、あとから読む人には分かりません。そのためマーカーには、残す語と一行の理由を一緒に書きます。

```markdown
<!-- term-drift:allow 結線 — 電気工事の文書なので本来の意味で使っている -->
```

これは lint で特定の指摘を対象外にするとき、その理由も残す考え方と同じです。現在のマーカーは出現箇所単位ではなく、**同じファイル内にある同じ語の全出現**を例外にします。その広い範囲を理解して置く必要があります。理由のないマーカーは例外として扱わず、別の語のマーカーが同じ行の検査を隠さないようにします。

## 履歴は書き換えず、現在の再発と分ける

コミットメッセージは、用語が導入された時期や文脈を知る有力な素材です。しかし公開済みの git 履歴を用語整理のために書き換えると、ハッシュと共同作業の前提を壊します。

term-drift はコミットを**発見の手掛かり**として走査しますが、適用対象にはしません。人が置換を承認した後も履歴だけに残る語は `historicalAcknowledged` として観測し、現在文書の未修正とは区別します。これにより、履歴を破壊せず、同じ過去の語を毎回新しい不具合として扱うループも避けます。

## 安全境界 — 外へ出さず、対象外へ書かない

対象文書を LLM API へ送る実装にすれば検出を製品内へ閉じられますが、リポジトリの内容を外部サービスへ送る責務と設定を背負います。term-drift はそれを選びません。インストール済み CLI の scan/apply 経路はネットワーククライアントを内蔵せず、検出は利用者が既に選んだ宿主エージェントの文脈で rules を読んで実行します。ただし、`npx` による導入・更新はパッケージ取得のため通信し得ます。また、宿主エージェントがどこへデータを送るかは term-drift CLI の保証範囲外です。

また、秘密ファイルは「読んでからマスク」するのではなく、収集段階で除外します。`.env`、鍵、認証情報らしいファイルとディレクトリを走査一覧へ載せません。走査素材内の指示文も命令ではなく、検査対象の外部由来テキストとして扱います。

ファイルシステムでも同じ境界を守ります。台帳や rules の symlink が対象リポジトリ外を指す場合は読み書きせず、`init` は対象内の通常ディレクトリだけへ配置します。対象外へ権限を広げない点は Saltzer & Schroeder の least privilege に対応します。秘密を収集段階で除外する点は、本ツールが採った収集範囲最小化の設計判断です。

## intent-planner との関係 — 意図のずれと語彙のずれ

intent-planner は、Intent・仕様・実装を互いに情報を落とす「損失付き射影」として扱います。term-drift は、その落差を**語彙の軸**で扱う独立ツールです。

意図を仕様へ写すとき、上流の言葉が別のラベルへ置き換わることがあります。仕様から実装へ進むとき、局所的な比喩がプロジェクト全体の概念名へ昇格することがあります。AI が生成した下流文書を次のAIが読むと、その言い換えが正規語彙であるかのように再生産されます。語彙のずれは、意味そのものが消える前に観測できる、小さな射影誤差です。

両ツールの接合は疎に保ちます。

- `.intent/glossary.md` があれば同じ台帳を正本として読む。
- term-drift が無いプロジェクトでは `.term-drift/glossary.md` を使う。
- 台帳の任意の分類列には、チーム共通語だけでなく承認済みの一般語判断も記録する。一般語判断は既定の読み手に対する語の分類だけを復元し、内輪転用や曖昧な文まで承認しない。
- 検出規則は term-drift の自然言語 rules を単一正本とし、他ツールは複製実装せず読む。
- runtime 依存や双方向同期を作らず、term-drift を依存チェーンの葉に保つ。

この分離により、intent-planner は問題空間と意図の整合を扱い、term-drift は既に広がった語彙の検出・判断権限・救済へ集中できます。

## 実測と限界 — 完全な検出器ではない

term-drift の検出規則は、opencode と hermes-agent の手動調査34件を使って有界に検証しました。検出対象として到達可能だった24件のうち、独立した検出エージェントが一致したのは17件、約71%でした。7件は見落とし、10件は走査素材に無いか識別子除外の設計上の対象外でした。詳細は [`test/fixtures/audits/l1a-comparison-20260712.md`](../test/fixtures/audits/l1a-comparison-20260712.md) にあります。

この値から言えるのは「完全に検出できる」ことではなく、むしろ次の限界です。

- 一般語と業界慣用語の境界は、LLMと人で割れる。
- 実在の固有名詞だと確信すると、説明なしでも除外しやすい。
- 到達可能な語でも単純に素通しすることがある。
- 正解リスト外の候補が、本当の新発見か過検出かは人の判定を要する。
- 宿主モデル、プロンプト、読む範囲によって結果が変わる。

したがって rules は真理を返す分類器ではなく、**人が確認すべき候補を作る検査手順**です。指摘ゼロは用語の完全性の証明ではありません。検査範囲を宣言し、見落としと構造的対象外を分け、効かなかった例も残す必要があります。

この不完全さが、意味判断の権限と決定的適用を分ける理由でもあります。検出の recall が不完全でも、一意な書き換え単位と再検査の安全性は独立して保てます。

## 設計姿勢 — 語彙を固定するのではなく、訂正可能にする

生きたプロジェクトでは、新しい概念と新しい語が必要になります。目標は新語を禁止することでも、全員に平易な語だけを強制することでもありません。

term-drift が守ろうとするのは、次の訂正可能性です。

- 新語には初見の人が辿れる説明がある。
- チーム共通語への昇格は人が判断する。
- 否認した語も履歴として残り、再発明に気づける。
- 意図的な例外には理由が残る。
- 誤った置換は git で戻せる。
- 検出規則は実測した見落としと過検出から更新できる。

語彙を静的な正解集合として凍結するのではなく、誰の言葉か、誰に通じるか、なぜ例外かを確認できる状態に保つ。それが term-drift の中心的な目的です。
