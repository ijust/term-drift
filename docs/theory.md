# term-drift の理論的背景

[English](theory_en.md) | 日本語

term-drift は、AI 支援開発を通じてプロジェクトに入り込んだ用語のずれを見つけ、**人が確認したか、明示的に任された範囲で低リスクと判断された言い換えだけ**を安全に適用するための軽量な仕組みです。

ここで扱う問題は、単なる表記ゆれではありません。同じ概念に複数の呼び名が生まれること、一般的な言葉がプロジェクト固有の意味で使われること、定義があっても使用箇所からたどり着けないことも含みます。こうした問題が積み重なると、文法的には読める文書でも、初めて参加した人には意味を理解できなくなります。

term-drift はこの問題を、Domain-Driven Design の **Ubiquitous Language（ユビキタス言語）**、用語管理、認知負荷、common ground（共有された理解）、情報探索、Human-in-the-Loop の考え方を組み合わせて扱います。意味の判断を完全には自動化せず、LLM、人、同じ入力なら同じ結果を返すプログラムに役割を分けるのが、設計の中心です。

**この文書を読まなくても term-drift は使えます。** 実際の手順は [`rules/workflow.md`](../rules/workflow.md) にあります。この文書は「なぜ台帳が必要なのか」「なぜ辞書との照合だけでは足りないのか」「なぜ誰の判断かを記録するのか」を知りたいときの参考資料です。

## 問題 — AI は「言葉を大量に持ち込む参加者」である

開発者同士でも用語は自然に揺れます。しかし AI コーディングエージェントは、短時間に大量の設計文書・コミットメッセージ・説明文を生成するため、揺れの**速度と複製範囲**を大きくします。

ここでいう **term drift** は、term-drift 自身への適用と、対象範囲を限定した調査で見つかった問題をまとめるために、本ツールが用いている概念です。Furnas らの vocabulary problem などは問題の一部を説明しますが、「AI が用語ずれを増幅する」という現象全体を直接測定した研究として引用しているわけではありません。

モデルは、一般的なコーパス、ほかのプロジェクトでの慣用表現、直前の会話から、もっともらしい語を選びます。しかし、その語が対象プロジェクトで合意されているかどうかは、言語モデルの語彙知識だけでは判断できません。その場では自然に見える表現でも、プロジェクト全体では次のようなずれを生むことがあります。

- 既存概念に別名を付け、検索と会話の入口を分裂させる。
- 一般語を特殊な意味に転用し、初見の読み手に二重の解釈負荷を与える。
- 一度だけ説明した語を参照なしで広げ、定義へ辿れない状態にする。
- 生成された語を次の生成が引用し、「多く使われているから正しい」という見かけを作る。

Furnas らの “The Vocabulary Problem” が示したように、人は同じ対象へ同じ語を選ぶとは限りません。AI はこの古典的な呼称不一致を消すのではなく、高い生成速度によって増幅し得ます。

term-drift が扱う **term drift** は、このような「語の形・意味・定義への経路が、チームの合意からずれること」です。識別子の改名や外国語の訳語選定一般は対象にしません。grep や実行手順で参照されるコマンド名・関数名・設定キーは機械的な契約でもあるため、散文の言い換えと混ぜないからです。

## 正規語彙は辞書ではなく、チームの社会的な契約である

Eric Evans の Ubiquitous Language は、専門家と開発者がモデルを語るために共有し、会話・文書・コードで継続して使う言葉です。重要なのは「一般の辞書に載っているか」ではなく、**このチームで同じ意味として合意されているか**です。

したがって term-drift は、語そのものの性質だけで分類しません。台帳の状態と想定する読み手に応じて、次の 3 つに分類します。

1. **一般語** — 想定した読み手の基礎語彙に入る語。
2. **チーム共通語** — 台帳に承認済みで載る語。
3. **未承認の独自用語の疑い** — 台帳に無い語、暫定の語、または一度否認されたのに再び使われた語。

この分類は、Clark & Brennan の grounding の考え方と一致します。伝達が成り立つには、送り手がその語を知っているだけでは足りません。相手も意味を理解できるという共通の基盤が必要です。「専門家なら分かる」「英語圏では普通」といった判断も、想定する読み手によって変わります。

台帳は真理の辞書ではなく、合意の記録です。そのため状態を持ちます。

- **承認済み** — チーム共通語として扱う。
- **暫定** — 誰かが使い始めたが、合意はまだない。
- **否認済み** — 採用しない。行は削除せず、同じ語の再発明を検出できるようにする。

状態が書かれていない語を承認済みとはせず、暫定として扱うのは、既存文書に出てきたというだけでチームが承認したことにはしないためです。

## なぜ3層で検出するのか

### 層1 — 台帳差集合

最初の層では、収集した文書に現れるプロジェクト固有らしい語を、台帳の正規語や別表記と照合します。これは controlled vocabulary を使った、基本的な用語管理です。

ただし、この層だけでは十分ではありません。台帳に無い固有名詞句は拾えても、`drain`、`settle`、`lobby`、`配線する` のように、辞書には普通の語として載る表現の特殊な使い方を見落とすからです。

### 層2 — 比喩転用と意味のずれ

Lakoff & Johnson が論じたように、比喩は飾りではなく、概念を理解し構造化する基本的な働きです。ソフトウェア開発でも `pipeline`、`drain`、`bridge` のような比喩は有用です。一方、比喩が局所的な意味を持ったまま説明なしで広がると、既知の一般的意味とプロジェクト内の意味を同時に保持させる二重負荷になります。

そのため第2層は、語が珍しいかではなく、**普通の意味からプロジェクト固有の意味へ転用されていないか**を読みます。これは単純な単語リストや正規表現では安定して判定できず、前後の文脈を読む層が必要になります。

### 層3 — 定義への到達可能性

語がどこかで定義済みでも、使用箇所からその定義へ辿れなければ、初見の読み手にとっては未定義とほぼ同じです。Pirolli & Card の information foraging では、人は手掛かり（information scent）を使い、読むコストに見合う情報源を探します。定義へのリンク・初出説明・用語集への参照がなければ、探索の手掛かりが弱くなります。

そこで第3層は、定義の**存在**だけでなく、使用箇所からの**到達可能性**を見ます。

| 状態 | 台帳差集合 | 比喩転用 | 到達可能性 |
|---|---:|---:|---:|
| 未登録の固有概念名 | 検出しやすい | 場合による | 定義があれば確認 |
| 一般語をプロジェクト固有の意味で使っている | 見落としやすい | 主に検出 | 定義への導線も確認 |
| 定義は別文書にだけ存在 | 登録済みなら沈黙し得る | 場合による | 主に検出 |

3層は競合する別方式ではなく、異なる見落とし方を補う関係です。

## なぜ検出を完全な機械判定にしないのか

「一般語か」「実在する外部の固有名詞か」「初めて読む人に説明が必要か」は、文字列だけでは判断できません。基礎語彙の固定リストを用意しても、言語、職種、プロジェクトごとの保守が新たな問題になります。Web 上での使用頻度を基準にすると、広く使われている語と、このチームで合意された語を混同してしまいます。

term-drift はここを LLM の読解へ委ねます。既定では人が判断し、利用者が明示した範囲では LLM が低リスク箇所を判断できます。

```text
走査対象の収集 ──> 文脈を読んだ検出・分類・提案 ──> 人による確認または明示的な委任
 再現可能なCLI              利用中のLLM                人／利用中のLLM
       │                                                   │
       └──────── 判断済み辞書 ──> 適用 ──> 再検査 <───────┘
                              再現可能なCLI
```

- **LLM** は候補と根拠を示し、明示的に任された範囲では低リスクな箇所を判断する。高リスクな箇所や意味を確定できない箇所は人に確認する。
- **人** は委任範囲を決め、重要な意味判断と台帳変更を担う。
- **CLI** は、誰の判断かが記録されたデータだけを、同じ入力なら同じ結果になる方法で適用する。

この役割分担は、Human-in-the-Loop を形だけ取り入れたものではありません。Bainbridge の “Ironies of Automation” が指摘するように、判断まで自動化するほど、人は監視に必要な注意力と文脈を失いやすくなります。そこで、重要な意味の判断と任せる範囲の決定は人が担い、その範囲内での低リスクな判断と、繰り返し行う適用処理だけを機械に任せます。

## 誰の判断かを記録し、意味が同じ箇所をまとめる

`approved: true` は、「モデルの確信度が高い」という印ではありません。人がその箇所を直接承認したか、明確な範囲をエージェントに任せ、その範囲で低リスクと判断された書き換えであることを示す **authorization（変更の承認根拠）** です。

新しい辞書では、この違いを自由記述の `note` だけに任せません。`decision_metadata_version: 1` を宣言し、各変更に `decision_source`、`decided_at`、`delegation_scope` を構造化して記録します。人が個別に承認した場合は `human-approved`、任された範囲でエージェントが判断した場合は `delegated-agent` です。CLI は適用前にこれらの組み合わせを検証し、適用結果にも同じ情報を返します。辞書を `.term-drift/` に残しておけば、本文にマーカーを混ぜず、誰の判断だったかを後から確認できます。以前の形式の辞書も互換性のため読み込めますが、判断した人は `legacy-unknown` と記録されます。

通常は、意味の同じグループごとに人が判断します。対象範囲を限定せず、すべての判断をエージェントに任せることは勧めていません。語によって、次のように必要な判断が異なるためです。

- 一般語としてそのまま残す。
- チーム共通語として台帳へ登録する。
- 普通の言葉へ言い換える。
- 特定のファイルだけ、理由を添えてそのまま残す。
- 意味を確認できず保留する。

利用者がリポジトリ、現在のレビュー、特定の語など、明確な範囲をエージェントに任せた場合、エージェントはすべての出現箇所を個別に読みながら、低リスクな箇所を判断できます。意味を確定できない箇所、重要な選択肢がある箇所、義務の強さや法務、セキュリティ、公開 API、実行時の挙動に関わる箇所は、人に確認します。これにより、確認の回数を減らしつつ、確認した箇所と適用範囲の記録は残します。

同じ語でも、使われている箇所によって、主体、対象、因果関係、表現の強さ、例外が異なります。すべての出現箇所を確認し、これらの意味、分類、書き換え方、元の意味を保てる理由が同じ箇所だけを同じグループにまとめます。ガイド方式での承認は、列挙した箇所だけに適用されます。エージェントに任せる方式では、指定された範囲内で後から見つかった箇所も、個別に確認したうえで低リスクならエージェントが判断できます。

通常の確認画面では、グループ内のすべてのファイルと行、各引用、書き換え後の文を列挙し、元の意味を保てる理由を 1 文で示します。代表例や件数だけを示して、承認対象を隠すことはありません。利用者はグループを分けたり、曖昧な箇所だけ詳しい説明を求めたりできます。承認後、CLI に渡す辞書では、グループを箇所ごとの項目に分け、対象の `path` と、そのファイル内で 1 箇所にだけ一致する文章を指定します。対話ではまとめて扱っても、適用時に語だけを一括置換しないためです。

台帳への登録も同じです。暫定から承認済みへの昇格は分類器の出力ではなく、チームの言葉を変更する意思決定だからです。

## 書き換えは翻訳ではなく、安全な移行作業である

用語の言い換えは小さな文字列の変更に見えますが、文書間の参照、コマンド例、識別子、過去の説明にも影響します。そのため term-drift は、書き換えをデータ移行に近い操作として扱います。

### 書き込み前の条件

- 対象は git 管理下で、ファイルが追跡済みである。
- git index と異なる、ステージされていない差分があっても、現在の内容に承認済みの `from` が一意に一致する場合は、対象外の差分を残したまま適用し、警告する。曖昧な一致や重なる書き換えは拒否する。
- UTF-8 として妥当でない文書は、再エンコードによるバイト破壊を避けるため触らない。
- Markdown の標準的なコードフェンス、単一行のインラインコード・HTML コメント・リンク先は機械参照として保護する。独自記法や壊れた Markdown まで完全に解析するパーサではない。
- 空の置換先、重複した置換元、連鎖して再適用時に結果が変わる辞書は拒否する。
- 新形式の辞書では、人承認または委任判断の構造化メタデータが欠ける項目を拒否する。
- 全対象を検証してから書き始め、辞書不正による途中適用を避ける。

事前検証により、不正な辞書を途中まで適用することは防げます。ただし、複数ファイルへの書き込みを 1 つのトランザクションとして処理しているわけではありません。書き込み中に I/O エラーやプロセス停止が起きても自動では元に戻らないため、その場合の復元には git を使います。

### 適用後の条件

- 承認した置換元が散文に残っていない。
- 再び適用しても変更が起きない（冪等）。
- 理由を添えて残したものだけが、例外として別枠に残る。
- 未検査・未適用の対象があれば成功扱いにせず、終了コードと `skipped*` で知らせる。

この構造は Design by Contract に近いものです。人が意味を承認したことが事前条件、適用後に残存がなく再適用で変化しないことが事後条件です。

## 例外として残すなら、理由も残す

どの用語にも例外はあります。電気工事の文書での「結線」のように、本来の意味で残すほうが正しい場合です。例外を禁止すると、利用者は検査そのものを無効化するようになります。

一方、理由を書かずに例外扱いすると、誤検出だったのか、判断を先送りしただけなのか、あとから読む人には分かりません。そのためマーカーには、残す語と一行の理由を一緒に書きます。

```markdown
<!-- term-drift:allow 結線 — 電気工事の文書なので本来の意味で使っている -->
```

これは lint で特定の指摘を対象外にするとき、理由も一緒に残す考え方と同じです。現在のマーカーは出現箇所ごとではなく、**同じファイル内にある同じ語の全出現**を例外にします。この広い適用範囲を理解したうえで置く必要があります。理由のないマーカーは例外として扱いません。また、別の語に対するマーカーが同じ行の検査を隠さないようにしています。

## 履歴は書き換えず、現在の再発と分ける

コミットメッセージは、用語が導入された時期や文脈を知る有力な素材です。しかし公開済みの git 履歴を用語整理のために書き換えると、ハッシュと共同作業の前提を壊します。

term-drift はコミットを**発見の手掛かり**として走査しますが、適用対象にはしません。人が置換を承認した後も履歴だけに残る語は `historicalAcknowledged` として観測し、現在文書の未修正とは区別します。これにより、履歴を破壊せず、同じ過去の語を毎回新しい不具合として扱うループも避けます。

## 安全境界 — 外へ出さず、対象外へ書かない

対象文書を LLM API に送る機能を組み込めば、検出処理を term-drift 単体で完結させられます。しかしその場合、リポジトリの内容を外部サービスへ送るための設定と責任も、term-drift が負うことになります。この設計は採用していません。インストール済み CLI の `scan` と `apply` はネットワーク通信機能を持たず、検出は、利用者がすでに選んでいるエージェントがルールを読んで実行します。ただし、`npx` による導入や更新では、パッケージを取得するために通信することがあります。また、利用中のエージェントがデータをどこへ送るかは、term-drift CLI が保証できる範囲には含まれません。

また、秘密ファイルは「読んでから隠す」のではなく、収集する段階で除外します。`.env`、鍵、認証情報らしいファイルやディレクトリは、走査対象の一覧に含めません。収集した文書に指示が書かれていても命令として実行せず、外部から渡された検査対象のテキストとして扱います。

ファイルシステムでも同じ境界を守ります。台帳やルールのシンボリックリンクが対象リポジトリの外を指している場合は読み書きせず、`init` は対象内の通常のディレクトリにだけファイルを配置します。対象外へ権限を広げない点は、Saltzer & Schroeder の least privilege に対応します。秘密情報を収集段階で除外するのは、本ツールが採用した、収集範囲を最小限に抑えるための設計です。

## intent-planner との関係 — 意図のずれと語彙のずれ

intent-planner は、Intent・仕様・実装を互いに情報を落とす「損失付き射影」として扱います。term-drift は、その落差を**語彙の軸**で扱う独立ツールです。

意図を仕様に落とし込むとき、もとの言葉が別の表現に置き換わることがあります。仕様から実装へ進むときには、一部で使われていた比喩が、プロジェクト全体の概念名として扱われることもあります。AI が生成した文書を別の AI が読むと、その言い換えがチームの共通語彙であるかのように繰り返されます。語彙のずれは、意味そのものが失われる前に観測できる、小さな変換誤差です。

両ツールの接合は疎に保ちます。

- `.intent/glossary.md` があれば、それを共通の台帳として読む。
- term-drift が無いプロジェクトでは `.term-drift/glossary.md` を使う。
- 台帳の任意の分類列には、チーム共通語だけでなく、一般語として承認した判断も記録する。一般語としての判断から復元できるのは、想定した読み手に対する語の分類だけである。その語をプロジェクト固有の意味で使っている箇所や、曖昧な文章まで承認したことにはしない。
- 検出規則は term-drift の自然言語ルールを唯一の参照元とし、ほかのツールは同じ処理を複製せずにそのルールを読む。
- runtime 依存や双方向同期を作らず、term-drift を依存チェーンの葉に保つ。

このように役割を分けることで、intent-planner は解決すべき問題と意図の整合性を扱い、term-drift は、すでに広がった語彙の検出、変更の判断、修正に集中できます。

## 実測と限界 — 完全な検出器ではない

term-drift の検出規則は、opencode と hermes-agent を手作業で調べて得た 34 件を使い、対象範囲を限定して検証しました。検出できる状態にあった 24 件のうち、別のエージェントによる検出結果と一致したのは 17 件で、約 71% でした。7 件は見落とし、残る 10 件は収集した文書に含まれていないか、識別子を除外する設計のため対象外でした。詳細は [`test/fixtures/audits/l1a-comparison-20260712.md`](../test/fixtures/audits/l1a-comparison-20260712.md) にあります。

この値から言えるのは「完全に検出できる」ことではなく、むしろ次の限界です。

- 一般語と業界慣用語の境界は、LLMと人で割れる。
- 実在の固有名詞だと確信すると、説明なしでも除外しやすい。
- 到達可能な語でも単純に素通しすることがある。
- 正解リスト外の候補が、本当の新発見か過検出かは人の判定を要する。
- 利用するモデル、プロンプト、読む範囲によって結果が変わる。

したがって、このルールは必ず正解を返す分類器ではなく、**人が確認すべき候補を挙げるための検査手順**です。指摘がゼロでも、用語に問題がないと証明されたわけではありません。検査した範囲を明らかにし、見落としたものと設計上の対象外を分け、検出できなかった例も残す必要があります。

この不完全さが、意味の判断と、書き換えを適用する処理を分けている理由でもあります。検出の再現率（recall）が完全でなくても、一意な範囲だけを書き換えることと、適用後にもう一度検査することの安全性は、別に保つことができます。

## 設計姿勢 — 語彙を固定するのではなく、訂正可能にする

生きたプロジェクトでは、新しい概念と新しい語が必要になります。目標は新語を禁止することでも、全員に平易な語だけを強制することでもありません。

term-drift が守ろうとするのは、次の訂正可能性です。

- 新語には初見の人が辿れる説明がある。
- チーム共通語への昇格は人が判断する。
- 否認した語も履歴として残り、再発明に気づける。
- 意図的な例外には理由が残る。
- 誤った置換は git で戻せる。
- 検出規則は実測した見落としと過検出から更新できる。

語彙を静的な正解集合として凍結するのではなく、誰の言葉か、誰に通じるか、なぜ例外かを確認できる状態に保つ。それが term-drift の中心的な目的です。
